# 作業指示書 05 ─ 合流・最終枝（product / vibe）

**前提:** 先に `00_共通仕様.md` を読むこと。
**この枝は他の全枝の“合流点”。** product は各枝の到達点を束ね、vibe はツリー全体の到達点。
**着手タイミング:** 各枝（特に Web / auto / AI・サーバー）の内容が固まってから書くと、参照の整合が取りやすい。
**書く順:** `product` → `vibe`

---

## product — まず一つ動かす（合流ハブ）
- ファイル: `lessons/product.html` ／ body: `lesson kind-common`
- 前提: static-deploy / auth / ai-eval / dataflow のいずれか1つに到達（＝どの枝から来てもよい）
- 次: read / git / devtools（品質枝へ分岐）
- kicker: `ONE WORKING THING`
- lead素材(desc): 選んだ範囲で、実際に一つ動くものを形にした証。完成範囲を小さく絞るほど、改善のサイクルを回しやすい。
- できるようになること(points): 最小の完成条件を満たしている／自分または他の人が同じ結果を再現できる
- AIへの言い方(SAY): 『今の欲しいものに対して過剰にせず、最小構成で。要らない技術は今は足さないで』
- 本文の核(quizヒント): 『他の人』『他の端末』が要らないうちはブラウザの中で足りる。技術が多いほど偉いわけではない、という“最小構成”の思想を中心に。例：自分・1端末・記録を残すだけなら HTML/CSS/JS + localStorage で十分。
- 補足: ここは「複数の枝が合流する地点」。各枝（静的公開・認証・AI評価・一括処理）のどれを通ってきても「一つ動くものができた」と言える、という束ね方を意識する。

## vibe — バイブコーディング（最終到達）
- ファイル: `lessons/vibe.html` ／ body: `lesson kind-final`
- 前提: iterate ／ 次: なし（最終）
- kicker: `FINAL MASTERY`
- lead素材(desc): AIに丸投げするのではなく、作りたいものが膨らむたびに『次に何が必要か』を自分で見極め、作り、確かめ、公開し、改善まで回せる状態。
- できるようになること(points): 目的から逆算してAIと協働できる／必要な技術を過不足なく選べる／小さく公開し、反応から次へ進める
- AIへの言い方(SAY): （専用なし）これまでの各単元の「AIへの言い方」を振り返り、総まとめとして提示する構成にする。
- 本文の方針: 新しい技術を教えるページではなく、**ツリー全体の総括**。「目的→必要な技術の見極め→作る→確かめる→公開→改善」のループを一枚で振り返らせる。各枝（web/auto/ai/server/quality）へのリンクや、これまでの単元を地図的に示すと締まる。
- lnav: 次の単元は無い（`<span class="next soon">…</span>` のまま、または「ツリーへ戻る」で締める）。
