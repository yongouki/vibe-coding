バイブコーディング ── ループを自分で回す

AIに丸投げするのではありません。作りたいものが膨らむたびに「次に何が必要か」を自分で見極め、作り、確かめ、公開し、改善まで回せる。ここまで来たあなたは、もうそれができます。

このツリーで身についたこと
  • 目的から逆算して、AIと協働で形にできる。
  • 欲しいものに対し、必要な技術を過不足なく選べる。
  • 小さく公開し、返ってきた反応から次の一手を決められる。

覚えるべきは技術ではなく、ループ

ここは新しい技術を教えるページではありません。これまで通ってきた単元を、一枚の地図として振り返る場所です。バイブコーディングの正体は、次の一つのループを止めずに回し続けることです。

目的を決める → 必要な技術を見極める → 作る → 確かめる → 公開する → 改善する。そしてまた目的へ戻る。新しい欲しいものが出てきたら、同じ輪を一周させるだけです。技術が増えても、回し方は変わりません。

ツリー全体を、ループに並べ直す

これまでの枝は、バラバラの知識ではなく、このループのどこかを担っていました。

  • 目的を決める（common）：完成条件を小さく決め、AIに文脈を渡し、タスクに分ける。
  • 必要な技術を見極める：他の人・他の端末・保存に加え、秘密・量・自動実行・停止影響を見て、web / auto / ai / server のどの道かを選ぶ。
  • 作る（各枝）：画面を出す、データを流す、APIや認証を立てる、AIの出力を扱う。
  • 確かめる・改善する（quality）：読む、動かす、記録する、直す、テストする、公開して、次へ反映する。
  • 一度締める（product）：過剰にせず、最小構成で「一つ動くもの」を再現できる形にする。

どの枝から来ても、最後は同じループの上に乗ります。順番に絶対の正解はなく、いま欲しいものに合わせて必要なところだけを通ればいい、というのがこのツリーの考え方でした。

AIへの言い方・総まとめ

各単元で出てきた「AIへの言い方」は、実はループの各段階に対応しています。専用の呪文を覚える必要はなく、いまループのどこにいるかで、渡す一文を選ぶだけです。

場面ごとの一文（これまでの総まとめ）

目的を決めるとき：「今日画面で確かめられる大きさまで小さくしてください。何をする・何が見える・今回はしない、の3点で書きます」

依頼するとき：「現状・今回の変更・触らない場所・確認方法を渡します。実装前に、どこをどう変えるか先に説明してください」

技術を選ぶとき：「今の欲しいものに対して過剰にせず、最小構成で。要らない技術は今は足さないでください」

確かめ・直すとき：「再現手順とエラー全文を渡します。原因の見立てを一つに絞ってから直してください」

改善するとき：「今回の反応を踏まえ、次に足す価値が高い変更を一つだけ提案してください」

共通しているのは、範囲を一つに絞り、確認方法を添え、実装の前に方針を言わせること。これさえ崩さなければ、どんな技術でもAIと安定して進められます。

よくある失敗

  • ループのどこにいるか曖昧なまま、いきなり「いい感じにして」と丸投げする。
  • 目的が膨らんだのに技術だけ足し続け、一度も「確かめる・公開する」へ進まない。
  • 公開して終わりにし、返ってきた反応を次の目的に戻さない。

ここだけ持ち帰る

  • バイブコーディングとは、目的→技術の見極め→作る→確かめる→公開→改善のループを回し続けること。
  • 各枝はバラバラの知識ではなく、このループのどこかを担っていた。
  • AIへの一文は、範囲を一つに絞り・確認方法を添え・先に方針を言わせる、で共通する。

最後にやってみる

いま一番作りたいものを思い浮かべ、ループを一周だけ言葉にしてください。一周回せたなら、もう次の欲しいものも同じように回せます。

目的：＿＿＿／必要な技術：＿＿＿／作る：＿＿＿／確かめる：＿＿＿／公開：＿＿＿／次の改善：＿＿＿

各欄に「何を見れば完了か」という証拠を一つ添えてください。作る前の予想、実行後の結果、失敗例を一つずつ残せれば、次の一周でも使える記録になります。
