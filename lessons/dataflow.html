Data processing

入力を読んで、変換して、安全に保存する

大量のデータをまとめて処理したいとき、画面は要りません。入力・変換・出力の3つに分けて考えると、最短で組めて、壊れにくくなります。

この単元でできるようになること
  • 処理を「入力・変換・出力」に分けて考えられる。
  • 途中経過と失敗した理由を記録して、後から原因を追える。

処理は、入力・変換・出力の3段で考える

「CSVを集計して」と一息に頼むと、どこで失敗したのか分からなくなります。最初に3つの段に分けておくと、問題が起きた場所がすぐ特定できます。

  1. 入力：元データを読む（ここでは絶対に書き換えない）
  2. 変換：集計や整形をする（途中の作業はコピーの上で）
  3. 出力：結果を保存する（最後にまとめて書き出す）

特に大切なのは、入力の元データを処理の途中で上書きしないことです。元が無事なら、失敗しても何度でもやり直せます。

元データを壊さない「置き換え方式」

1万行のような大きなデータでは、途中で止まる可能性を最初から織り込みます。直接本ファイルへ書き込むのではなく、一時ファイルに書いてから、全部うまくいったときだけ完成版へ差し替えます。入力元は、完成版とは別に残します。

  1. 先に1〜2行だけ試して、変換のルールが正しいか検証する
  2. 結果は一時ファイルへ書き出す（本ファイルにはまだ触らない）
  3. 全行が成功したら、一時ファイルを本ファイルへ置き換える

途中で止まっても、本ファイルは手付かずのまま。「開始前の状態へいつでも戻れる」状態を、設計で先に確保しておきます。

直書きで止まる 元のCSVに直接上書き → 5000行目でエラー → 半分書き換わったファイルだけが残る。
置き換え方式 一時ファイルへ書き出し → 全成功なら差し替え。失敗しても元データは無傷。

失敗した行があるときの方針を、先に決める

ここは、目的によって2つの正解があります。混ぜると「失敗行があるのに完成版を採用してよいのか」が曖昧になるので、処理前にどちらかを選びます。

  • 厳格モード：1行でも失敗したら完成版へ置き換えない。金額計算など、欠けた結果を使えない処理向け。
  • 許容モード：成功行を別の出力ファイルへ保存し、失敗行は行番号・理由とともに rejects.csv などへ分ける。入力元は置き換えない。

どちらでも、途中で何が起きたかをログへ残します。「止めないこと」より、部分的な結果を採用してよい処理かを先に判断することが大切です。

そのまま使える依頼文

入力を読んで変換して保存する流れにしてください。元データは変更しません。今回は厳格モードにし、一時ファイルへ出力して、全行成功したときだけ完成版へ置き換えてください。失敗時は置き換えず、行番号と理由をログへ残してください。

押さえるべきこと

  • 処理は入力・変換・出力に分け、入力の元データは絶対に上書きしない。
  • 先に検証 → 一時ファイル → 全成功時のみ完成版へ置き換える。
  • 失敗時は「全部不採用」か「成功行と失敗行を別出力」かを先に決める。

やってみよう

1万行のCSVを安全に変換する流れを、AIと一緒に設計してみましょう。次の文をそのまま渡してみてください。

1万行のCSVを別の形式に変換するスクリプトを設計してください。元データは変更せず、厳格モードと許容モードの違いを説明してください。今回は厳格モードで、一時ファイルへ書き、全行成功時だけ完成版へ置き換えます。わざと不正な1行を入れた場合に完成版が更新されず、行番号と理由がログへ残ることも確認してください。

チェックポイント

正常なCSVでは完成版が作られ、不正な1行を含むCSVでは元データと既存の完成版が変わらず、失敗理由がログへ残れば十分です。
