GIT

変更を「戻れる保存点」として残す

一つ動くものができたら、次はそれを安心して触れる状態にします。Gitは、変更を小さな保存点として積んでいく仕組みです。失敗しても前の点と比べて戻せるので、思い切って試せます。

今回の着地点

変更を目的ごとの保存点（コミット）に区切り、後から「この変更だけ戻したい」ができるようになる。

保存点があると、思い切って試せる

「ここまでは動いた」という状態を保存点として残しておくと、その後どれだけ壊しても、その点へ戻れます。Gitではこの保存点をコミットと呼びます。動くものができた直後は、最初のコミットを作る絶好のタイミングです。

保存点があれば、AIに大胆な変更を頼んでも怖くありません。気に入らなければ前の点へ戻すだけです。

分け方は「ファイル数」でなく「変更の理由」

コミットは、いくつのファイルを触ったかではなく、何のための変更かでまとめます。後で一つだけ取り消したくなったとき、これが効いてきます。

まとめてしまう バグ修正と色の変更とメモの追記を、1つのコミットに全部入れる。
理由で分ける 「追加ボタンの不具合を修正」「ボタンの色を変更」を、別々のコミットにする。

右側なら、「色の変更だけやっぱり戻したい」という判断がしやすくなります。Gitでは、コミット済みの変更を打ち消す新しいコミットを作る git revert が使えます。ただし、後の変更がそのコミットへ依存している場合は競合したり、追加の修正が必要になったりします。

「戻す」の種類を混ぜない

  • コミット済みの変更を打ち消す：共有した履歴では、基本的に git revert で反対の変更を新しいコミットとして残す。
  • まだコミットしていないファイルを戻す：差分を確認してから git restore を使う。未保存の変更が消えるので慎重に行う。

操作前には git status と git diff で、いま何が変わっているかを確認します。「戻したい」とだけAIへ伝えず、コミット済みか、未コミットかも添えます。

コミットの言葉は「何を変えたか」が分かるように

各保存点には短い説明（コミットメッセージ）を付けます。未来の自分が読んで分かる言葉にします。

後で読めない update / fix / いろいろ修正
変更が分かる 空欄のとき追加されない不具合を修正

AIに保存点づくりを手伝ってもらう

Gitの操作に慣れていなくても、AIへ目的ごとに分けるよう頼めます。

そのまま使える依頼文

この変更をGitで保存点として残したいです。まずgit statusとgit diffを見せ、目的ごとにコミットを分ける案を説明してください。私が確認してからコミットし、それぞれ何を変えたか分かるメッセージを付けてください。戻す場合は、コミット済みならgit revert、未コミットなら差分を確認してからgit restoreを検討し、実行前に消える変更を説明してください。

ここだけ持ち帰る

  • 「動いた」状態は保存点（コミット）として残す。戻れるから試せる。
  • コミットはファイル数でなく、変更の理由ごとに区切る。
  • メッセージは、後で読んで何を変えたか分かる言葉にする。
  • 戻す前にstatusとdiffを確認し、revertとrestoreを使い分ける。

5分だけやってみる

練習用の小さなリポジトリで、いま動いている状態を最初のコミットとして保存します。次に色だけを変えて別コミットにし、git log と git diff HEAD~1 で内容を確認してください。最後にその色変更を git revert で打ち消し、履歴と画面の両方を確認します。
