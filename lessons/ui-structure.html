Components / State / Routing

画面が増えても、追える形に整理する

画面や状態が増えて、一枚のコードでは追いにくくなったら整理の合図です。部品（Component）・状態（State）・ページ遷移（Routing）の3つに分けて、見通しを取り戻します。

今回の着地点

UIを役割ごとのComponentに分け、状態の置き場所とページ遷移（Routing）を決められるようになる。

分けるのは「大きくなって追えなくなってから」でいい

最初から部品に分ける必要はありません。1ファイルで動くうちは、それが一番速いからです。整理を考えるのは、こういう兆候が出てきたときです。

  • 1つ直すと、関係ないところまで壊れる
  • 同じような表示を、あちこちにコピーして書いている
  • どの値が今どうなっているのか、自分でも追えなくなってきた

これは設計の失敗ではなく、作るものが育った証拠です。育ったぶんだけ、整理し直せば先へ進めます。

3つに分けて考える

整理の軸は、保存場所の話ではありません。「画面の責任」と「状態の持ち主」を決めることです。次の3つに分けます。

  1. Component（部品）：画面を役割ごとの部品に切る。結果表示は結果表示、履歴は履歴。
  2. State（状態）：どの値を、どの部品が持つかを決める。共有する値は1段上に置く。
  3. Routing（ページ遷移）：画面の切り替えを、URLや画面の状態として整理する。
追えなくなる 全部の表示と値を1ファイルに詰め、どこが何を持っているか分からないまま足し続ける。
責任を分ける 表示を部品に切り、共有する値の持ち主を1か所に決めてから足していく。

フレームワークへ移すかは、分けた「後」で決める

整理＝Reactなどへの移行、ではありません。Vanilla JS（素のJavaScript）のままでも、関数やファイルを役割ごとに分ければ見通しは良くなります。フレームワークが効いてくるのは、部品や画面が十分に増えてからです。

だからAIに頼むときも、いきなり移行を前提にせず「Vanilla JSを続ける案」と並べて比較してもらうと、過剰にならずに済みます。

AIへ頼むときは「責任の分け方」を聞く

1ファイルが大きくなってきました。役割ごとの部品（Component）に分けて、状態（State）の置き場所とページ遷移（Routing）を整理してください。まだフレームワークへ移行すべきかも、Vanilla JSを続ける案と比較してください。

「状態の置き場所」「Vanilla JSを続ける案と比較」と添えるのがポイントです。これで、ただ書き直すのではなく、責任の分け方とその理由を一緒に出してもらえます。

ここだけ持ち帰る

  • 分けるのは「一枚では追えなくなってから」で十分。
  • 軸は保存場所ではなく、画面の責任（Component）と状態の持ち主（State）。
  • フレームワーク移行は整理の後で判断。Vanilla JS継続案と比べて決める。

やってみよう

運勢アプリを、結果表示・履歴・設定の3つの部品に分ける設計図を作ってみましょう。下の文をそのままAIへ渡せます。

現在は1ファイルのHTMLとJavaScriptで運勢アプリを作っています。結果表示、履歴、設定の責任を分けるComponent構成と、どこにStateを置くかを提案してください。まだフレームワークへ移行すべきかも、Vanilla JSを続ける案と比較してください。
