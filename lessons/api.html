API / Secrets

外部サービスを、安全につなぐ

外部サービス（AIなど）へリクエストするとき、APIキーのような秘密をコードから切り離す必要があります。鍵はブラウザに置けません。どこに置き、どう経由させるかを覚えます。

今回の着地点

APIの入力・出力・エラーの約束を読み、秘密の鍵はサーバー側・環境変数に置いて、ブラウザから直接漏らさない構成にできるようになる。

APIは「約束ごと」でできている

外部サービスを呼ぶとは、相手が決めた約束に従って頼むことです。約束には、だいたい3つの面があります。これを読めれば、つなぐ準備は半分できています。

  • 入力：何を、どんな形で送るか（質問の文、画像、など）
  • 出力：何が、どんな形で返るか（多くはJSON）
  • エラー：断られたとき、どんな番号や理由で返るか

http の単元で見た「リクエストとレスポンス」と同じ往復です。違いは、多くのAPIが鍵（APIキー）を要求すること。この鍵の扱いが、安全につなぐかどうかの分かれ目になります。

鍵はブラウザに届いた時点で「見られる」

ここが一番のつまずきどころです。ブラウザに送られたものは、変数名をどう工夫しても、利用者が中身を見られます。だから鍵をブラウザ側のコードに書くと、公開した瞬間に誰でも使えてしまいます。

ブラウザから直接呼ぶ 画面側のコードに鍵を書いて、ブラウザからAIのAPIを直接呼ぶ。鍵が丸見えになり、勝手に使われる。
バックエンド経由で中継 鍵はサーバー側に隠し、ブラウザは自分のバックエンドにお願いするだけ。鍵はブラウザに渡さない。

正しい流れはこうです。ブラウザ → 自分のバックエンド → 外部API。鍵を使う処理はバックエンドの中だけで起こり、利用者の手元には届きません。そして鍵そのものはコードに直接書かず、環境変数（コードの外に置く設定値）に入れます。

AIへは「鍵はサーバー側・環境変数」と言う

頼み方に一言入れるだけで、危ない構成を避けられます。

APIキーはブラウザに置かず、自分のバックエンド経由で外部APIを呼ぶ形にしてください。鍵は環境変数に入れ、コードに直接書かないでください。ブラウザからはバックエンドにお願いするだけにし、入力・出力・エラーの扱いも説明してください。

「ブラウザに置かず」「バックエンド経由」「環境変数」。この3語が、鍵を守るための合言葉です。

ここだけ持ち帰る

  • APIは入力・出力・エラーの約束。まずそれを読む。
  • ブラウザに届いたものは誰でも見られる。鍵は絶対に置かない。
  • 流れは「ブラウザ→自分のバックエンド→外部API」。鍵は環境変数へ。
