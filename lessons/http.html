HTTP / JSON

外のデータを呼んで、画面に出す

画面やプログラムが外部サービスとやり取りするときの共通語がHTTPです。こちらから送る「リクエスト」と、返ってくる「レスポンス」、そしてその中身であるJSONの読み方を覚えます。

今回の着地点

公開APIを呼んで、返ってきたJSONの必要なキーだけを取り出して画面に出す。読み込み中と失敗時の表示まで作れるようになる。

やり取りは「送る・返る」の往復

外のデータを使うとは、誰かが用意したサービスへ「これください」とお願いし、答えを受け取ることです。お願いがリクエスト、答えがレスポンス。この往復がHTTPです。難しい仕組みを覚えなくても、まずはこの2語を区別できれば十分です。

  • リクエスト：どのURLに、何を、どう頼むか（猫の画像を1枚ください、など）
  • レスポンス：頼んだ結果。成功したか、そして中身（データ）が返る

返ってくる中身は、多くの場合 JSON という形です。キー: 値 が並んだ、住所録のようなデータだと思ってください。欲しいのは全部ではなく、その中の特定のキーの値だけ、ということがほとんどです。

JSONは「キーで値を取り出す」

たとえば猫画像APIが、こんなJSONを返したとします。

[
  {
    "id": "abc123",
    "url": "https://example.com/cat.jpg",
    "width": 640,
    "height": 480
  }
]

このとき画面に出したいのは url の値だけ、というのはよくあること。「JSON全体をどう扱うか」ではなく、「どのキーの値が欲しいか」をAIに伝えると、必要な部分だけを取り出してもらえます。

AIへは「呼ぶURL・出すキー・失敗時」をセットで

通信は、成功するときばかりではありません。電波が切れる、サービスが混んでいる、頼み方を断られる。だから最初から「読み込み中」と「失敗したとき」の表示も一緒に頼むと、現実で動くものになります。

この公開APIをfetchで呼んで、返ってきたJSONのこのキーを画面に出してください。読み込み中の表示と、失敗したときの表示も入れてください。1ファイルのHTMLで動かし、変更点も説明してください。

成功しか考えない APIを呼んで結果を表示して、とだけ頼む。回線が切れると画面が固まったまま無反応になる。
失敗も画面に出す 読み込み中はその表示を出し、失敗したら「読み込めませんでした」と出す、まで頼んでおく。

「届かない」と「断られた」は別の話

うまくいかないとき、原因は大きく2種類あります。そもそも通信できていないのか、通信はできたが中身を断られたのか。これを混同すると、直す場所を間違えます。

  • 通信できていない：URLが違う、回線が切れている。送る前後の問題。
  • 通信後に断られた：届いたが、頼み方が不正だと返された（たとえば 422 のような番号で返る）。中身の問題。

レスポンスには成否を表す番号（ステータス）が付いてきます。AIに「失敗時はどちらなのか分かるように表示して」と頼むと、原因の切り分けがぐっと楽になります。

fetchは404や500でも、自動では失敗にならない

fetch() が例外になるのは、主に回線やURL形式などでレスポンスを受け取れなかったときです。サーバーから404や500が返った場合はレスポンス自体を受け取れているため、response.ok を自分で確認します。

const response = await fetch(url);
if (!response.ok) {
  throw new Error(`HTTP ${response.status}`);
}
const data = await response.json();

もう一つ、ブラウザにはCORSという制限があります。相手のサーバーがブラウザからの利用を許可していないと、サーバーが応答していてもJavaScriptから中身を読めません。「ネットワーク」「HTTPステータス」「CORS」を分けて確認します。

ここだけ持ち帰る

  • 外のデータは「リクエストを送り、レスポンスを受け取る」往復で扱う。
  • JSONは欲しいキーの値だけを取り出せば十分。全部を扱おうとしない。
  • 「届かない」と「断られた」は別。失敗時の表示を最初から頼んでおく。
  • fetch後は response.ok を確認する。404や500は自動では例外にならない。
