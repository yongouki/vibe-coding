Server / Runtime

ブラウザの外で、コードを動かす

ブラウザの中だけでは足りなくなる瞬間があります。データを皆で共有したい、秘密の情報を隠したい。そのとき必要になるのが、ブラウザの外でコードを動かす場所＝サーバーです。

今回の着地点

「どこでコードが動くか」を意識し、ブラウザに置けない処理をサーバー側へ移す判断ができるようになる。

コードが動く場所は1つではない

これまで書いてきたコードは、利用者のブラウザの中で動いていました。それで足りるうちは、それが一番簡単です。けれど、次のような願いが出てきた瞬間に、ブラウザの中だけでは無理が出ます。

  • みんなで同じデータを見たい（自分の端末の中だけでは共有できない）
  • APIキーのような秘密を隠したい（ブラウザに置くと誰でも見られる）
  • 誰が来ても同じ処理を、確実に走らせたい

こうした処理は、利用者の端末ではなく、自分が管理する「外の場所」で動かします。それがサーバーです。コードを書くこと自体より、その処理がどこで動くべきかを考えられることが、ここでの肝心な点です。

ブラウザに置けないものを、外へ移す

判断はシンプルです。利用者に見られて困るもの、利用者ごとにバラバラだと困るものは、ブラウザに置かない。サーバー側へ移します。

ブラウザに置いてしまう APIキーを画面側のコードに直接書く。公開すると、誰でもその鍵を読み取れてしまう。
外へ移す 鍵を使う処理はサーバー側に置き、ブラウザからはサーバーにお願いするだけにする。

「秘密を隠す」「データを共有する」が出てきたら、それはサーバーを足すサインです。逆に、まだ自分の端末の中で完結しているなら、無理に外へ出す必要はありません。

AIへは「何のために外へ出すか」を言う

サーバーの作り方は1つではありません。だから「外で動かす目的」を伝えると、AIが目的に合った最小構成を選んでくれます。

APIキーを隠したい（または、データを共有したい）ので、ブラウザの外（サーバー側）で動かす最小構成にしてください。まずは一番小さく動く形で作り、起動方法も説明してください。

ここで覚えておきたいのは、Node.jsは選択肢の一例にすぎないということです。PythonやBaaS（バックエンドを借りる仕組み）、Serverless（必要なときだけ動く仕組み）も、要求次第で候補になります。「これしかない」と決めつけず、目的を伝えて選んでもらいましょう。

ブラウザからサーバーへ届く値は、利用者が自由に書き換えられます。サーバー側では「画面で入力制限したから安全」と考えず、型・長さ・許可する値をもう一度検証します。秘密を隠すことと、届いた入力を信用しないことが、サーバー側へ移すときの二つの基本です。

ここだけ持ち帰る

  • コードが「どこで動くか」を意識する。ブラウザの中か、外か。
  • 秘密を隠す・データを共有する、が出たらサーバーを足すサイン。
  • Node.jsは一例。Python・BaaS・Serverlessも目的次第で候補になる。
