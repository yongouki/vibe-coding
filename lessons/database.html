端末を越えて、データを残す

たくさんのデータを、端末をまたいで保存し、後から探し出せる場所がデータベースです。「他の人とも共有したい」「履歴を残したい」が出てきたら、その出番です。

今回の着地点

保存したい項目とその関係を決め、localStorageとデータベースの違い（共有・検索）を理解して使い分けられるようになる。

localStorageは「その端末の中」にしか残らない

これまで使ってきた localStorage は手軽ですが、保存先はあくまでその人のブラウザの中です。便利な反面、こんな限界があります。

  • 別の端末で開くと、データが無い（同じ人でもスマホとPCで共有されない）
  • 他の人からは、その内容を見られない（共有できない）
  • 「条件で探す」「並べ替える」が苦手（検索に向かない）

つまり「自分の、この端末の中だけ」で完結するうちは localStorage で十分。けれど、共有や検索が欲しくなった瞬間に、保存する場所そのものを変える必要が出てきます。それがデータベースです。

まず「何を・どんな形で残すか」を決める

データベースに移すときに一番大事なのは、仕組みより保存する項目を先に決めることです。支出メモなら、たとえばこう。

支出（1件ぶん）
  ├ 金額   … 数値
  ├ メモ   … 文字
  ├ 日付   … いつの記録か
  └ 持ち主 … 誰の記録か（共有するなら必要）

項目と、その関係（誰の記録か、など）がはっきりしていれば、後の検索や共有がやりやすくなります。先に全部を完璧に決める必要はありません。今欲しい項目から始めて、後で足せます。

AIへは「共有したいからDBへ移す」と言う

「保存して」とだけ言うと、AIは手軽な localStorage を選びがちです。端末をまたいで残したい意図と、残す項目を伝えると、データベースを使う構成にしてもらえます。

いまlocalStorageに保存している支出メモを、端末をまたいで残したいので、localStorageではなくデータベースに保存する形へ移してください。保存する項目は金額・メモ・日付・持ち主です。まずは一番小さく動く構成で作り、変更点も説明してください。

場所を言わない 「データを保存できるようにして」とだけ頼み、結局その端末の中だけに残る形になる。
場所と項目を言う 「端末をまたいで残したいのでDBへ。項目は金額・メモ・日付・持ち主」と伝える。

ここだけ持ち帰る

  • localStorageは「その端末の中」にしか残らない。共有・検索には向かない。
  • 端末をまたいで残す・探す・共有する、が欲しくなったらデータベース。
  • 移すときは、まず保存する項目とその関係を先に決める。

やってみよう

いまlocalStorageに残しているデータを1つ思い浮かべ、「端末をまたいで共有したくなる場面」を1つ書き出してください。そのうえで、保存したい項目を3〜4個、上の図のように並べてみましょう。これがそのままAIへの依頼の土台になります。
